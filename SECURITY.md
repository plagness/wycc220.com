# Политика безопасности

## Как сообщить об уязвимости

Не создавайте публичный issue. Используйте **GitHub Private Vulnerability Reporting** в разделе Security репозитория. До включения функции не публикуйте технические детали и свяжитесь с владельцем репозитория через его GitHub-профиль.

Мы подтвердим получение, оценим влияние и согласуем раскрытие. Добросовестное исследование без доступа к чужим данным и нарушения работы приветствуется.

## Секреты

- секреты находятся только в менеджере секретов окружения;
- frontend получает только явно публичные переменные;
- production, preview и development используют разные ключи с минимальными правами;
- ключи сторонних платформ не выдаются контрибьюторам и не попадают в логи;
- обнаруженный в Git секрет считается скомпрометированным: сначала отзыв и ротация, затем очистка истории;
- CI проверяет изменения на секреты, зависимости и опасные паттерны.

Локальные значения хранятся в `.env`, который игнорируется Git. `.env.example` содержит только названия и безопасные заглушки.

## Границы доверия

Фанатские игры, арты и внешние ссылки считаются недоверенными. Они не получают cookies основного домена, серверные токены или прямой доступ к базе. Загружаемые файлы проходят проверку типа, размера и вредоносного содержимого и отдаются с отдельного домена без credentials.

Подробная модель угроз: [docs/SECURITY-MODEL.md](./docs/SECURITY-MODEL.md).
