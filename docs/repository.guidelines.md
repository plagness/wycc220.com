# Ведение Git и файлов

Этот документ задаёт единые правила репозитория. Они нужны, чтобы множество независимых разработчиков могли работать предсказуемо и не создавать скрытых рисков для production.

## Имена файлов и каталогов

Имена пишутся на английском языке, строчными буквами. Смысловые части составного имени разделяются точкой:

```text
game.manifest.json
data.policy.md
security.model.md
user.activity.service.ts
video.card.test.tsx
```

Дефис и подчёркивание не используются как разделители в новых проектных файлах:

```text
# Правильно
game.manifest.json
repository.guidelines.md

# Неправильно
game-manifest.json
repository_guidelines.md
манифест-игры.json
```

Имя должно описывать назначение, а не автора, временный статус или номер личной задачи. Запрещены имена вроде `max.fix.ts`, `new.file.ts`, `final.v2.ts` и `temp.ts`.

### Технические исключения

Внешний инструмент может требовать точное имя. Такие файлы не переименовываются произвольно:

- `README.md`, `LICENSE`, `SECURITY.md`, `CONTRIBUTING.md`, `CODEOWNERS` — community-файлы, автоматически распознаваемые GitHub;
- `CODE_OF_CONDUCT.md`, `PULL_REQUEST_TEMPLATE.md`, `ISSUE_TEMPLATE` — имена и каталоги GitHub;
- `package.json`, `pnpm-lock.yaml`, `pnpm-workspace.yaml`, `tsconfig.json` — соглашения Node.js, pnpm и TypeScript;
- `next-env.d.ts` — генерируется и подключается Next.js;
- `.env.example`, `.gitignore`, `.editorconfig` — стандартные dotfiles.

Новое исключение допускается только со ссылкой на документацию инструмента в pull request.

## Размещение файлов

- deployable-приложения находятся в `apps/<name>`;
- переиспользуемый код и контракты — в `packages/<name>`;
- архитектурные документы — в `docs`;
- значимые архитектурные решения — в `docs/adr`;
- исходный код — в `src`, тест располагается рядом как `name.test.ts`;
- сгенерированные файлы, сборки, логи и локальные секреты не коммитятся.

Нельзя создавать общую папку `utils` без чёткой предметной области. Предпочтительнее `media.url.ts`, `activity.score.ts` или отдельный модуль домена.

## Ветки

В upstream существуют только постоянные `dev` и `main`:

- `dev` принимает обычные pull request;
- `main` отражает production и обновляется релизным PR из `dev`;
- рабочие ветки создаются в fork контрибьютора и удаляются после merge;
- прямой push и force push в `dev` и `main` запрещены;
- один pull request решает одну связанную задачу.

## Коммиты

Коммит должен быть небольшим, законченным и проходить проверки. Сообщение оформляется на английском в формате Conventional Commits:

```text
feat(games): add project manifest validation
fix(api): limit activity page size
docs(repository): define file naming rules
test(activity): cover duplicate chat events
chore(deps): update fastify
```

Допустимые основные типы: `feat`, `fix`, `docs`, `test`, `refactor`, `perf`, `build`, `ci`, `chore`, `revert`. Нельзя смешивать форматирование всего проекта с функциональным изменением.

## Pull request

Перед открытием PR:

1. Обновить ветку от актуальной `dev`.
2. Просмотреть `git diff` и исключить случайные файлы.
3. Убедиться, что нет `.env`, токенов, дампов, персональных данных и больших бинарных файлов.
4. Выполнить `pnpm format:check` и `pnpm check`.
5. Объяснить пользовательский результат, риски и способ отката.
6. Для API, данных, рейтинга или границ безопасности добавить ADR.

Нельзя просить ревью, если PR заведомо не собирается. После замечаний автора review нельзя скрывать историю обсуждения переписыванием смысла PR.

## Секреты и бинарные файлы

Секрет никогда не коммитится, даже в удалённый позже файл. Если это произошло, ключ немедленно отзывается и ротируется до очистки истории.

Арты, модели, аудио и видео не добавляются в Git без утверждённой политики хранения. Для крупных ресурсов используется объектное хранилище; в Git остаются манифест, лицензия, контрольная сумма и источник.

## Ответственность за изменения

Автор PR отвечает за понятность и исходную проверку. Reviewer отвечает за корректность границ, безопасность и соответствие хартии. Мейнтейнер, выполняющий merge, отвечает за состояние целевой ветки, но не получает авторских прав на вклад участника.
