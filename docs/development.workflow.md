# Рабочий процесс команды

## Выбор задачи

Issue должен содержать результат для пользователя, границы, направление команды и критерии готовности. Для неизвестной архитектуры сначала создаётся короткий research/ADR issue, а не большой PR с необратимым решением.

## Вертикальный срез

Предпочтителен небольшой законченный путь: контракт → API → UI → тест → документация. Большая функция делится по independently mergeable этапам, которые не оставляют небезопасный endpoint или неиспользуемый сбор данных.

## Интеграция

Работа ведётся в fork, PR направляется в `dev`. Изменение одного направления требует review его владельца; общий контракт, безопасность или CI требуют дополнительного профильного review. Релиз из `dev` в `main` содержит changelog, миграции и план отката.

## Definition of ready

- понятен пользовательский сценарий;
- определена ответственная команда;
- известны затрагиваемые данные и внешние API;
- перечислены acceptance criteria;
- для рискованного изменения согласован ADR.

## Definition of done

- код находится в правильном модуле;
- публичные контракты обновлены до потребителей;
- тесты, типы, lint и build проходят;
- документация и примеры актуальны;
- проверены безопасность, доступность и приватность по уровню риска;
- нет секретов, случайных binary assets и неиспользуемых зависимостей;
- PR содержит откат и влияние на rollout.
