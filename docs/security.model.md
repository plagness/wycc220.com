# Модель безопасности

## Активы

Ключи YouTube/Twitch/Telegram, база пользователей и событий, учётные записи модераторов, DNS и deployment, репутация проекта, работы авторов и доступность публичного API.

## Основные угрозы и меры

| Угроза                             | Базовая мера                                                                           |
| ---------------------------------- | -------------------------------------------------------------------------------------- |
| Секрет в коммите или логе          | deny-by-default CI, gitleaks, маскирование, ротация, отдельные ключи окружений         |
| Вредоносный PR из fork             | workflow без секретов и write-token, review владельца компонента, pinned actions       |
| XSS из арта или игры               | отдельный origin, запрет SVG/HTML по умолчанию, CSP, sandbox iframe                    |
| Захват аккаунта модератора         | passkey/2FA, минимальные роли, журнал действий, два одобрения для production           |
| Накрутка рейтинга                  | idempotency, rate/anomaly limits, версионируемые правила, ручная апелляция             |
| Деанонимизация пользователя        | только добровольная привязка аккаунтов, opt-out, публичные агрегаты                    |
| Нарушение правил внешней платформы | официальные API, минимальное хранение, обработка удалений, документированный retention |
| Supply-chain атака                 | lockfile, Dependabot, dependency review, минимальное число зависимостей                |
| Злоупотребление публичным API      | rate limit, quotas, cache, безопасные лимиты выборки, abuse contact                    |

## Правила CI

- workflows из PR имеют только `contents: read`;
- `pull_request_target` запрещён для запуска кода из PR;
- production job работает только из `main` и защищённого environment;
- токены GitHub получают минимальные permissions;
- сторонние actions фиксируются на полном commit SHA перед включением production deploy;
- результаты сканирования не должны печатать найденный секрет целиком.

## Реакция на секрет в истории

1. Немедленно отозвать ключ у провайдера.
2. Выпустить новый ключ с меньшими правами.
3. Проверить журналы использования и остальные окружения.
4. Удалить значение из актуального кода и при необходимости переписать историю.
5. Сообщить контрибьюторам о необходимости обновить клоны.
6. Зафиксировать причину и профилактическую проверку без публикации самого секрета.
